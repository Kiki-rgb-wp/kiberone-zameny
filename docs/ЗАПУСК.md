# Пошаговый запуск «Замены KIBERone»

## Шаг 1. Supabase — проект и база

1. Зайди на [supabase.com](https://supabase.com), войди в аккаунт.
2. **New project** — создай проект (укажи пароль к БД и регион).
3. В проекте открой **Project Settings → API** и скопируй:
   - **Project URL** (например `https://xxxx.supabase.co`)
   - **anon public**
   - **service_role** (секретный ключ — никому не показывай)

## Шаг 2. Применить миграции в Supabase

**Вариант А — через веб-интерфейс (проще всего):**

1. В Supabase открой **SQL Editor**.
2. Открой файл в проекте: `supabase/migrations/20240306000001_initial_schema.sql`, скопируй весь текст и выполни в SQL Editor (**Run**).
3. Затем открой `supabase/migrations/20240306000002_rls_policies.sql`, скопируй и выполни.

**Вариант Б — через Supabase CLI:**

```bash
# Установка CLI (если ещё нет): https://supabase.com/docs/guides/cli
npm install -g supabase

# В корне проекта
cd "киьер заменв"
supabase login
supabase link --project-ref ТВОЙ_PROJECT_REF   # ref смотри в URL проекта в дашборде
supabase db push
```

## Шаг 3. Пользователь и роль админа

1. В Supabase открой **Authentication → Users**.
2. **Add user → Create new user** — введи email и пароль (это твой аккаунт для входа).
3. Открой **SQL Editor** и выполни (подставь свой email):

```sql
UPDATE public.profiles SET role = 'admin' WHERE email = 'ТВОЙ_EMAIL@example.com';
```

Если строки в `profiles` ещё нет (пользователь создан вручную, а не через регистрацию), сначала создай запись:

```sql
INSERT INTO public.profiles (id, email, role)
SELECT id, email, 'admin' FROM auth.users WHERE email = 'ТВОЙ_EMAIL@example.com'
ON CONFLICT (id) DO UPDATE SET role = 'admin';
```

## Шаг 4. Переменные окружения бэкенда

1. В папке проекта скопируй пример env-файла:

```bash
cd backend
cp ../.env.example .env
```

2. Открой `backend/.env` и заполни (значения из шага 1):

```
SUPABASE_URL=https://ТВОЙ_PROJECT.supabase.co
SUPABASE_ANON_KEY=eyJ...   # anon public
SUPABASE_SERVICE_ROLE_KEY=eyJ...   # service_role
```

Остальные переменные (Resend, Telegram и т.д.) можно не трогать для первого запуска.

## Шаг 5. Запуск бэкенда

```bash
cd backend
npm install
npm run dev
```

В консоли должно появиться: `API запущен на порту 3000`. Оставь этот терминал открытым.

## Шаг 6. Запуск фронтенда

Открой **второй** терминал:

```bash
cd "путь/к/проекту/киьер заменв/frontend"
npm install
npm run dev
```

Vite покажет что-то вроде: `Local: http://localhost:5173/`

## Шаг 7. Вход в приложение

1. В браузере открой **http://localhost:5173**
2. Введи email и пароль пользователя, которого создал в шаге 3.
3. После входа откроется главный экран: расписание, заявки, кнопка «Найти замену».

## Регистрация новых преподавателей

**Самoregистрация (рекомендуется):** на экране входа есть ссылка **«Регистрация»**. Преподаватель вводит email, пароль и при желании ФИО, нажимает «Зарегистрироваться». Ему создаётся аккаунт с ролью **tutor** (триггер в БД). После этого он может войти и видеть своё расписание, создавать заявки на замену и откликаться на чужие.

**Имя в профиле:** если при регистрации указано ФИО, оно сохранится в `profiles.name` (нужна миграция `20240306000004_profile_name_from_metadata.sql` — выполни её в SQL Editor, как шаг 2).

**Подтверждение email:** если в Supabase включено «Confirm email», после регистрации пользователю придёт письмо; без подтверждения войти нельзя, пока не подтвердит или админ не отключит эту настройку в Authentication → Settings.

## Если что-то не работает

- **401 при запросах** — проверь, что в `.env` верные `SUPABASE_URL` и ключи; что пользователь есть в Authentication.
- **Нет расписания** — для админа можно добавить смены через API (POST /schedule с токеном) или позже через UI. Для преподавателя расписание пустое, пока админ не создаст записи в `schedule`.
- **CORS / сеть** — фронт должен ходить на `http://localhost:5173`, а запросы к API идут через прокси Vite на `http://localhost:3000`; оба сервера должны быть запущены.
